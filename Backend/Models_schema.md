# Database Schema & Sync Logic

Архитектура Rentify использует модель **Distributed SQLite to Central PostgreSQL**. 
Для обеспечения консистентности данных при синхронизации используется поле `local_sqlite_id`.

## Основные сущности (ER Diagram)

### 1. User & Access
* **CustomUser**: Главный аккаунт владельца бизнеса. Хранит лимиты устройств (`device_limit`) и статус подписки.
* **SubAccount**: Конкретная точка проката. Имеет привязку к железу (`hwid`) и уникальный `api_token`.

### 2. Business Logic
* **Client**: Данные клиентов. Поля `iin` и `bin` индексированы для быстрого поиска по базе eGov.
* **Catalog**: Модели оборудования (описание, общая цена, залог).
* **Inventory**: Физические единицы. Имеют поле `status` (available, rented, broken, archived) и `next_maintenance_date`.
* **Rent**: Заголовок сделки. Связывает клиента и точку проката. Статусы: `draft`, `booked`, `active`, `overdue`, `closed`.
* **RentItem**: Конкретные инструменты в аренде с "замороженной" ценой и тарифом на момент сделки.
* **Payment**: Транзакции. Поддерживают частичную оплату и разделение `amount_rent` / `amount_consumable`.

## Логика синхронизации
Каждая таблица на сервере имеет уникальный индекс:
`unique_together = ('owner', 'local_sqlite_id')`

Это позволяет клиенту генерировать ID локально в офлайн-режиме, а серверу при синхронизации выполнять `update_or_create`, не опасаясь дублей или конфликтов ID разных пользователей.
